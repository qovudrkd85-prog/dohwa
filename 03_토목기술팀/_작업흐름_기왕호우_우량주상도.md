# 작업 흐름 — 기왕호우·우량주상도 조사 (토목기술팀)

> 이 파일은 "기왕호우·우량주상도 조사" 작업의 **표준 흐름**이다.
> 사용자가 트리거 문장을 말하면 01 → 03 직원이 이 순서대로 분담 실행하고,
> 최종 결과물은 `00_result/`에 저장한다.

## 폴더 구조

```
03_토목기술팀/
├── 00_result/                팀 최종 결과물 (작업 1건 = YYYY-MM-DD_HH-MM_요청명/)
├── 01_data-download-desk/     직원 1 — 자료 다운로드
└── 03_storm-rainfall-desk/    직원 3 — 분강우량·우량주상도 워크북 생성
```

## 트리거

> "토목기술팀, **○○ 관측소 기왕호우·우량주상도 분석해줘** —
>  2010.08.18, 2011.08.08, 2012.08.30 …"

입력값: **관측소명**, **호우 일자 목록**(사상별 중심일), **프로젝트명**(선택).

---

## 1단계 — 01_data-download-desk · 자료 다운로드

KMA 기상자료개방포털(AWS 방재기상관측, `pgmNo=56`)에서 사상마다 2종을 받는다.

- **전제**: 9222 디버그 크롬 실행 + data.kma.go.kr 로그인 상태.
- **분강우(분자료)** — 자료형태 **분 자료**. 분자료는 **하루 단위**로만
  받히므로, 사상 1건당 **중심일 전·당·후 3일**을 각각 따로 받는다.
  (예: 중심일 2011.08.08 → 08.07, 08.08, 08.09 세 번)
- **시강우(시간자료)** — 자료형태 **시간 자료**. 사상 범위(3일)를 한 번에 받는다.
- **저장(중간물)**: `01_data-download-desk/업무/YYYY-MM-DD_<관측소>_기왕호우우량주상도/`
  에 받은 원본 + `다운로드기록.md`. 가공하지 않는다.

### ⚡ AWS 원샷 스크립트 (포털 왕복 단축)

`01_data-download-desk/aws_portal_script.js` — 포털 조작을 함수 3개로 묶음.
ASOS와 같은 포털 엔진이라 같은 방식으로 쓴다:
1. `navigate` → `selectAwsRltmList.do?pgmNo=56`
2. STEP1 — 지점 선택 (`STATION` 교체)
3. STEP2 — 자료형태·기간·요소·조회 (`DATAFORM`/`START`/`END` 교체).
   분 자료는 하루씩 → 사상당 STEP2+STEP3를 3일 반복.
4. STEP3 — 다운로드 + 용도신청(토목/건축).
5. `OBS_AWS_*` 파일 회수.

### 🔍 단계별 2차 검증 (필수)

| 단계 | 검증 | 실패 시 |
|---|---|---|
| STEP1 | `checked`에 관측소명 있음 | 멈춤·재시도 |
| STEP2 | `stnIds` 채워짐 + `searched:true` | 멈춤·재시도 |
| STEP3 | `biHweonWarning:false` | 재로그인 |
| 다운로드 파일 | 분자료=해당 1일치 / 시간자료=3일치, 날짜 일치 | 부분·오류 의심 멈춤 |
| 03 워크북 | build_minute/build_hyetograph 정상 종료, 사상수 일치 | 재실행 |
| 최종 | `00_result`에 워크북2 + 원본 + 작업일지 | 보완 |

→ 각 단계를 검증 통과한 것만 다음으로 넘긴다.

## 2단계 — 03_storm-rainfall-desk · 워크북 생성

- **입력**: 1단계가 받은 분강우·시강우 xls.
- 받은 파일을 자기 업무 폴더에 **사상별 폴더**로 정리한다:
  ```
  03_storm-rainfall-desk/업무/YYYY-MM-DD_<관측소>_기왕호우우량주상도/
  ├── 분강우/01_<관측소>_2011.08.08/  (08.07·08·09.xls)
  └── 시강우/01_<관측소>_2011.08.08/  (시강우.xls)
  ```
- CivilTools 검증 모듈 호출:
  ```
  storm_rainfall_desk.py build_minute     <업무>/분강우  00_분강우량_<관측소>.xlsx
  storm_rainfall_desk.py build_hyetograph <업무>/시강우  <프로젝트>_(<관측소>)…(우량주상도).xlsx  <프로젝트>
  ```
- → ① 분강우량 워크북(10·60분 최대강우 수식), ② 우량주상도 워크북(차트).
- **최종 결과물**: `00_result/YYYY-MM-DD_HH-MM_<요청명>/` 에 워크북 2개 +
  원본자료 복사본 + 작업일지를 함께 넣는다(2차 검증용).

## 결과물

`00_result/YYYY-MM-DD_HH-MM_<요청명>/` 안의 **분강우량 워크북 + 우량주상도 워크북**.

## ⚠️ 수정사항 (2026-05-18 — 은척 작업에서 확정)

### 다운로드 — 강수량만
시강우(시간자료)·분강우(분자료) 모두 **요소는 "강수"만** 선택한다.
전체 요소를 받으면 컬럼이 과다해 오류 → `aws_portal_script.js`는
요소 트리에서 "강수" 그룹만 체크하도록 수정됨.

### 우량주상도 — 템플릿 채우기 (build_hyetograph 폐기)
우량주상도는 03 모듈로 새로 만들지 않는다. **기존 템플릿 엑셀을 채운다.**

- 템플릿(참고본): `E:\배평강\2025 project\2025_project\2025_지평지구\
  02_기초자료\07_실강우량 조사_우량주상도\2026.05.18_우량주상도_re\
  지평지구_(은척)집중호우시 강우조사(우량주상도) .xlsx`
- 구조: **사상별 시트 1장**, 각 시트에 **차트가 이미 셀에 묶여 있음**.
  - A열 = 날짜시간(datetime, 1시간 간격)
  - B열 = 강수량(시간값, mm)
  - C열 = 누가강수량(수식 — 이미 들어있음)
  - 데이터는 **4행부터**.
- 작업: 템플릿을 복사 → 각 사상 시트 A4↓·B4↓에 시강우 데이터를 붙여넣기.
  **강우 시작 = 강수량이 처음 0이 아닌 시각부터** (사용자 확정: b안).
  C열·차트는 자동 갱신.
- **차트 바인딩 = 4행부터 고정 행수** (사상01 검증: A4:A46/B4:B46 = 43행).
  강우 시작점부터 그 행수만큼 붙여넣는다("그래프에 맞게").
  ✅ 검증 2026-05-18 — 사상01 다운로드 시강우 = 템플릿 sheet1 43행 0불일치.
- 차트 보존: openpyxl은 차트를 떨어뜨릴 수 있음 → win32com(실제 Excel) 사용.

### 우량주상도 채우기 — 검증된 스크립트 (숙달용)
`03_storm-rainfall-desk/fill_hyetograph.py` — 시강우 xls를 템플릿 시트에
한 줄로 채운다 (win32com, 차트·수식 보존). 검증 2026-05-18 사상01.
```
py fill_hyetograph.py <시강우.xls> <템플릿.xlsx> <시트번호>
```
- 시트번호: 사상01→1, 02→2, 03→3, 04→4
- 강우 시작(첫 비0)부터 차트 바인딩 행수만큼 자동으로 채움
- 사상마다 이 명령 1번 → 4개 사상 = 4번 실행이면 끝
- build_hyetograph_workbook(새 워크북 생성)은 이 작업에 쓰지 않는다.

### 분강우량 워크북
`storm_rainfall_desk.py build_minute` 는 그대로 사용 (검증됨).

## 직원 분담 원칙

- 01은 **받기만**, 03은 **정리·만들기만**. 서로 영역 침범 안 함.
- 01이 받은 파일을 03이 입력으로 받는 릴레이. 최종물은 00_result로.
- 토목·설계 판단(설계강우 적용 등)은 L3 — 사람이 결정한다.
